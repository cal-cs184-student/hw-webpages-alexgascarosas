Overview

In this homework, we implemented several geometric modeling techniques related to Bezier curves, Bezier surfaces, and triangle meshes. Overall, the assignment focused on understanding how smooth curves and surfaces can be represented mathematically and how triangle meshes can be manipulated using the half-edge data structure.

In the first section, we implemented Bezier curve evaluation using de Casteljau’s algorithm. The idea behind the algorithm is to repeatedly linearly interpolate between control points using a parameter t. Each step creates a smaller set of intermediate points until only one point remains, which lies on the Bezier curve. We implemented the function that performs one step of this interpolation and then used it recursively to evaluate the curve. Next, we extended this idea to Bezier surfaces. Instead of evaluating a single curve, the surface evaluation works by applying the same 1D algorithm in two directions. First, each row of control points is treated as a Bezier curve and evaluated using parameter u. This produces a set of points that form another curve, which is then evaluated using parameter v to get the final point on the surface.

In the second part of the homework, we worked with triangle meshes using the half-edge data structure. We implemented area-weighted vertex normals, which are used for smooth shading. To compute this, we iterated over all faces connected to a vertex, computed each face’s normal using a cross product, weighted it by the face area, and summed them together before normalizing the result. After that, we implemented two mesh operations: edge flips and edge splits. Edge flipping changes the diagonal shared by two triangles without adding or removing elements, while edge splitting inserts a new vertex at the midpoint of an edge and creates additional edges and faces. These parts required carefully updating many pointers in the half-edge structure to make sure all vertices, edges, faces, and halfedges still pointed to the correct elements. Finally, we implemented loop subdivision to upsample the mesh. This algorithm subdivides each triangle into four smaller triangles and updates vertex positions using weighted averages of neighboring vertices. I first computed the new positions for vertices and edge midpoints, then performed edge splits and flips to subdivide the mesh, and finally updated the vertex positions.

One thing we found interesting about homework 2 is how much detail is involved in working with meshes. The math concepts like interpolation and subdivision are fairly straightforward, but implementing them correctly with the half-edge data structure required careful debugging and attention to how everything is connected.

Section I: Bezier Curves and Surfaces

Part 1: Bezier curves with 1D de Casteljau subdivision

De Casteljau's algorithm takes a list of control points and a scalar parameter t in [0, 1], and recursively produces a new list of points where each point lies on the line segment between two consecutive points from the previous level. The precise position is determined by t: when t = 0 the point coincides with points[i], and when t = 1 it coincides with points[i+1]. In my implementation, I loop from index 0 to (size - 2), linearly interpolating each consecutive pair (points[i], points[i+1]) to build the next level, then repeat this process on successive levels until a single point remains.

Part 2: Bezier surfaces with separable 1D de Casteljau

De Casteljau's algorithm extends to Bezier surfaces by applying it separably along each parameter axis. Given an n x n grid of control points and two parameters u and v, I first run the full 1D algorithm along each row at parameter u, reducing each row of n points down to a single point. This produces n new points, one per row, which together define a new Bezier curve running in the other direction. I then run the full 1D algorithm on those n points at parameter v to obtain the final surface point P(u, v). In my implementation, evaluate1D repeatedly calls evaluateStep until only one point remains, and evaluate iterates over each row of controlPoints, calling evaluate1D with u to collect the n intermediate points, then calls evaluate1D again on those points with v to produce the final result.

Section II: Triangle Meshes and Half-Edge Data Structure

Part 3: Area-weighted vertex normals

For Part 3, I implemented area-weighted vertex normals by looping through all the triangles connected to a given vertex using the half-edge structure. Starting from the vertex’s halfedge, I kept moving around the neighboring faces with twin()->next() until I got back to the starting halfedge. For each non-boundary face, I took the three triangle vertices and formed two edge vectors from the current vertex to the other two vertices. Then I used the cross product of those two vectors to get the face normal. Since the magnitude of that cross product is proportional to the triangle’s area, this automatically gives an area-weighted normal, so I could just add it directly to the running sum. After going through all neighboring faces, I normalized the final sum to make it a unit normal vector. This gives a smoother normal at the vertex compared to just using one face normal, which is why the teapot looks smooth with Phong shading instead of having the more faceted flat-shaded look. One thing I had to be careful about was making sure the normals pointed outward instead of inward, since otherwise the shading looked way too dark. I also had to use HalfedgeCIter since normal() is a const function and shouldn’t modify the mesh.

Part 4: Edge flip

For the edge flip implementation, the goal was to change the diagonal shared by two adjacent triangles. Given two triangles (a,b,c) and (c,b,d), flipping the edge (b,c) replaces it with the edge (a,d), which changes the triangles to (a,d,c) and (a,b,d). First, I checked if the edge or either of its neighboring faces was on the boundary. If it was, I just returned the edge without flipping it since boundary edges should not be flipped. After that, I gathered all the halfedges, vertices, edges, and faces involved in the two triangles around the edge. The main part of the implementation was updating the halfedge connectivity. I reassigned the next, twin, vertex, edge, and face pointers using setNeighbors() so that the halfedges correctly represented the new triangle configuration after the flip. I also updated the halfedge pointers stored in the vertices, faces, and the flipped edge to make sure everything still pointed to valid elements in the mesh.

One debugging trick that helped a lot was drawing the two triangles before and after the flip and labeling all the halfedges. This made it much easier to keep track of which pointers needed to change. Another thing I did while debugging was repeatedly flipping the same edge multiple times in the viewer to make sure the mesh stayed consistent and didn’t break or create holes. During debugging I initially had issues where some faces would render incorrectly or parts of the mesh would disappear. This usually meant that one of the halfedge pointers was assigned incorrectly. Carefully checking every pointer and comparing it to my diagram helped me find the mistakes. Once all the pointers were set correctly, the edge flipping worked consistently across the mesh.

Part 5: Edge split

For edge split, given an edge (b,c) shared by triangles (a,b,c) and (c,b,d), I insert a new vertex m at the midpoint of (b,c) and create 4 new triangles: (a,c,m), (a,b,m), (b,d,m), (c,d,m). I allocate 8 new halfedges, 4 new edges, 4 new faces, and 1 new vertex, then set all neighbor pointers via setNeighbors for every halfedge in the region, including the 4 border halfedges whose next and face pointers change. After wiring everything up, I delete the old edge, its two halfedges, and the two old faces, rather than reassigning them.

For an implementation trick, I listed out every relevant mesh element upfront and named them directly after the diagram shown in lecture, which made the setNeighbors calls straightforward to write and verify against the picture. I also chose to delete old elements instead of reassigning pointers, which added a few extra lines but made the logic easier to follow.

Debugging was tedious since the counter clockwise winding order tripped me up a lot since my brain naturally goes clockwise, so I kept getting next pointers backwards. Most of the debugging was manually checking each setNeighbors call one by one against the diagram. Drawing the before and after diagrams with labeled halfedges was essential.

Part 6: Loop subdivision for mesh upsampling

For loop subdivision, I followed the general steps described in the assignment. First, I computed the new positions for all the original vertices and stored them in newPosition without immediately updating their actual positions. For each vertex, I looped through its neighboring vertices using the half-edge structure, counted how many neighbors it had, and applied the Loop subdivision formula to compute the updated position. Next, I computed the positions for the new vertices that would be created when splitting edges. For each edge, I used the weighted formula involving the two endpoints of the edge and the two opposite vertices from the adjacent triangles. I stored this value in Edge::newPosition. After that, I split all the original edges in the mesh. To make sure I didn’t accidentally split edges that were created during the process, I first stored the original edges in a vector and only iterated through those. When a new vertex was created during the split, I marked it as isNew and set its position to the value I had already computed earlier. Then I flipped the edges that connected one old vertex and one new vertex. This helps produce the correct triangulation pattern for Loop subdivision. Finally, after all the topology changes were done, I copied the newPosition values into the actual vertex positions.

One thing that helped during debugging was doing the steps in order and checking them one at a time. If something looked wrong in the mesh, it was easier to figure out whether the issue came from the vertex position calculations, the edge splitting, or the edge flipping. Keeping track of isNew for vertices and edges was also really important, because without that it was easy to accidentally flip or process the wrong edges. When running loop subdivision a few times on meshes, I noticed that sharp edges and corners get smoothed out pretty quickly. The mesh starts to look more rounded because the vertex positions are basically being averaged with their neighbors. Even shapes with sharp corners will gradually lose those corners after a few iterations. When I tested loop subdivision on cube.dae, the cube became slightly asymmetric after a few iterations. This happens because the cube is made of triangles, and the way each square face is split into triangles isn’t perfectly symmetric. Since Loop subdivision works on the triangle connectivity, small differences in how the triangles are arranged can cause the smoothing to behave slightly differently in different areas. To make the cube subdivide more symmetrically, I tried flipping some edges first so that the triangulation was more balanced across the faces. After doing that, the subdivision results looked more even. Changing the connectivity beforehand helps the algorithm behave more symmetrically.
